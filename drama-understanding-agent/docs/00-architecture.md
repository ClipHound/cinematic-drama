# 00 - 总体架构设计

> Drama Understanding Agent v1.0
> 定位：以 Project 为隔离边界、以 Doubao-Seed-2.0-lite 全模态模型为理解核心、以结构化剧情记忆为状态管理、以 Action Plan 驱动工具执行的多集短剧连续理解 Agent。

---

## 0.1 一句话定义

> 输入一部短剧的全部视频文件，Agent 按剧集顺序逐集观看理解，持续构建和演化角色档案、剧情事件、人物关系、伏笔线索等结构化记忆，最终输出可供下游消费的完整剧情知识库。

## 0.3 核心数据流（单集处理）

```
输入: episode_video.mp4 + ASR结果(可选) + 累积记忆上下文

    ┌─────────────────────────────────────────────────────┐
    │ Step 1: 构建 Prompt                                  │
    │   - 已知角色卡摘要 (来自记忆)                         │
    │   - 未解决伏笔列表 (来自记忆)                         │
    │   - 上集摘要 (来自记忆)                               │
    │   - 当前集 ASR 文本 (如果有)                          │
    │   - 指令: 输出 Action Plan                           │
    └──────────────────────┬──────────────────────────────┘
                           │
                           ▼
    ┌─────────────────────────────────────────────────────┐
    │ Step 2: 模型调用                                     │
    │   - 将视频 + 文本 Prompt 发送给 Doubao Seed 2.0 Lite │
    │   - 模型返回结构化 Action Plan (JSON)                 │
    └──────────────────────┬──────────────────────────────┘
                           │
                           ▼
    ┌─────────────────────────────────────────────────────┐
    │ Step 3: Action Plan 解析与执行                        │
    │   - 解析 JSON 为 Action 列表                         │
    │   - 按依赖顺序执行每个 Action                         │
    │   - 写类 Action → 生成 State Patch                   │
    │   - 读类 Action → 查询记忆返回结果                    │
    │   - 资产类 Action → ffmpeg 截帧/截片段                │
    └──────────────────────┬──────────────────────────────┘
                           │
                           ▼
    ┌─────────────────────────────────────────────────────┐
    │ Step 4: State Patch 提交                             │
    │   - full_auto 模式: 高置信直接提交，冲突标记           │
    │   - hitl 模式(未来): 暂停等待人工审核                  │
    │   - 提交后记忆系统持久化更新                          │
    └──────────────────────┬──────────────────────────────┘
                           │
                           ▼
    ┌─────────────────────────────────────────────────────┐
    │ Step 5: 集间过渡                                     │
    │   - 生成本集摘要 (由模型在 Action Plan 中输出)        │
    │   - 更新全局剧情状态                                  │
    │   - 快照当前记忆状态 (便于回滚)                       │
    │   - 进入下一集循环                                    │
    └─────────────────────────────────────────────────────┘
```

## 0.4 设计决策记录 (ADR)

### ADR-1: 视频直接输入，不做帧网格
- **决策**: 利用 Doubao Seed 2.0 Lite 的原生视频理解能力，每集直传完整视频
- **理由**: 从实验结果看（5分钟视频约 47K tokens，耗时 50-59s），质量远优于帧网格方案；成本不是约束
- **后果**: 依赖 Doubao 视频 API 可用性；单次调用 timeout 需设为 120s+

### ADR-2: 单次调用输出完整 Action Plan
- **决策**: 不做多轮对话，一次调用让模型输出包含所有操作的 Action Plan
- **理由**: 减少 API 调用次数；模型有全局视野时规划更好；符合你的「模型负责理解，系统负责治理」原则
- **后果**: 输出可能很长（需 max_tokens 8192+）；需要 JSON 容错解析

### ADR-3: SQLite + Qdrant 双存储
- **决策**: 结构化数据用 SQLite（角色、事件、关系等），语义检索用 Qdrant
- **理由**: SQLite 零部署依赖、事务可靠、便于快照回滚；Qdrant 提供向量相似度检索用于角色匹配和事件查询
- **后果**: 需要保持两者一致性；State Patch 提交时需原子操作

### ADR-4: 角色跨集识别策略 = 模型语义判断 + embedding 相似度
- **决策**: 模型判断新出现的角色是否为已知角色，系统用角色描述的 embedding 相似度作为辅助验证
- **理由**: 短剧角色称呼多变（"角色A/公子A/那个纨绔"）；纯文本匹配不够；embedding 可以捕捉语义等价
- **后果**: 需要在 Prompt 中注入已知角色列表；可能有误合并，full_auto 模式下自动合并高相似度、标记低置信度

### ADR-5: Project 物理隔离
- **决策**: 每部剧一个独立目录，包含独立的 SQLite 文件、Qdrant collection、资产文件夹
- **理由**: 彻底防止跨剧污染；便于删除/归档单个项目；便于并行处理多部剧
- **后果**: 不支持跨剧知识共享（也不需要）

### ADR-6: 先 full_auto，HITL 作为扩展点预留
- **决策**: MVP 只实现 full_auto 模式，但在 State Patch 层预留 `review_required` 标记
- **理由**: 快速验证端到端流程；HITL 只需在 Patch 提交前加一个审核阻塞即可
- **后果**: 可能有错误记忆累积；通过快照+回滚机制缓解

## 0.5 技术栈

| 层级 | 选型 | 理由 |
|:---|:---|:---|
| 语言 | Python 3.11+ | 生态、现有代码基础 |
| 模型 | Doubao-Seed-2.0-lite (火山方舟) | 原生视频理解、质量验证通过 |
| 结构化存储 | SQLite (via sqlmodel) | 零部署、事务、快照 |
| 向量存储 | Qdrant (本地模式) | 成熟、支持过滤、有 Python SDK |
| Embedding | BGE-M3 或 Qwen3-embedding (本地) | 中文语义质量好 |
| ASR | 复用现有 Whisper/FunASR | 已有基础设施 |
| 视频处理 | ffmpeg (subprocess) | 截帧、截片段 |
| 数据模型 | Pydantic v2 | 类型安全、JSON Schema 自动生成 |
| CLI | typer | 简洁、类型注解驱动 |
| 测试 | pytest | 标准选择 |

## 0.7 与现有系统的对接

本 Agent 系统取代 pipeline 中的理解阶段，产出物兼容下游消费：

```
现有 pipeline:
  ASR → Audio → [Understand] → Highlight → Orchestrate → Gate → Manifest → Expansion

重构后:
  ASR → [Drama Understanding Agent] → (产出 understanding.json 兼容格式)
                                     → Highlight → Orchestrate → ...
```

Agent 的 `output/` 目录中的数据可以直接转换为现有 `understanding.json` 格式，确保下游 `s3_highlight.py` 等模块无需改动。
