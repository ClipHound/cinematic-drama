# 短剧交互组件库精确描述标准

> 版本: 2.0
> 用途: 供组件库维护方与 LLM 交互设计 Agent 对齐
> 基于: 第一轮实际运行中发现的语义漂移、标签粒度过粗、组件误选等问题

---

## 一、标准总则

### 1.1 组件定义必须包含的五个部分

每个组件的描述须包含以下五个部分，缺一不可：

| 编号 | 部分 | 作用 | 缺失后果 |
|------|------|------|---------|
| P1 | **核心语义定义** | 一句话定义这个组件在观众心理层面的本质 | LLM 无法区分相近组件 |
| P2 | **适用场景正例** | 列出 "应该选这个组件" 的具体场景类型，附判断标准 | LLM 只做关键词匹配，不做场景推理 |
| P3 | **硬排除规则** | 列出 "绝对不能选这个组件" 的场景条件 | IP5 问题：team_cheer 用于正邪对立场景 |
| P4 | **选择优先级** | 当多个组件同时满足 P2 条件时，谁优先 | LLM 随意选择一个，缺乏决策依据 |
| P5 | **正例与反例** | 至少 2 个正例 + 2 个反例，附推理说明 | LLM 无法校准判断边界 |

### 1.2 情绪标签与组件的映射

组件库不再使用单一 `emotion` 字段做匹配，改为 **"场景叙事功能"为主、"观众心理状态"为辅** 的双维度定义：

```
选择流程:
  第一步: 判断场景的叙事功能类别 (见 §1.3)
  第二步: 在对应类别内，按观众心理状态选择具体组件
  第三步: 逐条检查 P3 硬排除规则
  第四步: 如多个组件候选，按 P4 优先级决策
```

### 1.3 场景叙事功能分类体系

所有短剧场景按叙事功能分为 6 类，每个组件归属其中一类：

| 类别 | 定义 | 包含组件 |
|------|------|---------|
| **情绪宣泄** | 观众积累的某种情绪需要释放出口 | shatter_strike, anger_release, tear_resonance, laugh_burst |
| **立场表达** | 观众需要在冲突中选择或表达态度 | team_cheer, guardian_shield |
| **认知参与** | 观众需要动脑思考、推理或预判 | prediction_card, clue_judge_card |
| **氛围渲染** | 场景本身的氛围值得标记或庆祝 | celebrate_confetti, sugar_storm |
| **节奏过渡** | 情绪段落之间的缓冲衔接 | emotion_buffer |
| **集尾结构** | 固定出现在集尾的结构性互动 | episode_end_prediction |

**LLM 选择组件时，必须先判断场景叙事功能类别，再在类别内选择具体组件。** 这避免了 "funny → laugh_burst" 的浅层关键词匹配——先问"这个场景是情绪宣泄类吗？"如果不是（比如是认知判断而非情绪释放），laugh_burst 就不能选。

## 二、十二个组件的精确定义

以下每个组件按 P1-P5 五部分完整定义。P3 硬排除规则是强制性的——只要触发任何一条，该组件就不可选。

---

### 2.1 shatter_strike（碎屏暴击）

**P1 核心语义定义**：
观众积压的不满/愤怒/期待获得**即时且彻底的爽感释放**。核心心理动作是"终于！"——之前有明确的情绪积累，此刻一次性清空。

**P2 适用场景正例**：
- (a) 反派被打脸：之前有反派嚣张/欺压的铺垫，此刻遭到报应
- (b) 主角反击成功：主角在隐忍/压抑后爆发展现实力
- (c) 关键反转落地：长期伏笔在此刻揭晓并带来满足感
- 通用判断标准：**此场景之前必须有 ≥1 个场景积累了同向情绪（愤怒/憋屈/期待），且此刻是那个情绪的释放点**

**P3 硬排除规则**：
- [EX-01] 场景本身是情绪积累（铺垫）而非释放——此时应选 emotion_buffer 或不设互动点
- [EX-02] 场景中的"爽"来自悲剧性事件（如反派被杀的快感伴随主角的道德堕落）——此时应选 tear_resonance
- [EX-03] 场景的情绪主导是"好笑"而非"解气"——此时应选 laugh_burst

**P4 选择优先级**：
- shatter_strike > celebrate_confetti：当爽感来自"击败对手"而非"达成目标"时
- anger_release > shatter_strike：当观众还需要继续愤怒（情绪未到释放点）时

**P5 正例与反例**：
- ✅ 正例：蛮夷嚣张半集后被神秘高手隔空秒杀——前半集积累的愤怒在此刻一次性清空
- ✅ 正例：主角被诬陷后当众拿出关键证据翻转局面
- ❌ 反例：反派第一次出场嚣张——此时应该积累愤怒（anger_release 或不设点），而非释放
- ❌ 反例：主角复仇成功后陷入道德困境——爽感不纯粹，不适用 shatter_strike

---

### 2.2 anger_release（生气宣泄）

**P1 核心语义定义**：
观众对场景中的不公正/欺压/憋屈产生**共情性愤怒**，需要一个表达愤怒的出口。核心心理动作是"太过分了！"——观众和角色同步感到被冒犯。

**P2 适用场景正例**：
- (a) 角色被不公正对待：被冤枉、被欺压、被背叛
- (b) 反派做出令人发指的行为：观众产生道德愤怒
- (c) 系统性的憋屈：角色长期处于不利地位，观众累积不满
- 通用判断标准：**场景的核心情绪是"这不对/这不公平"，且观众需要一个情绪出口来表达这种不满**

**P3 硬排除规则**：
- [EX-01] 场景的情绪已经到达释放点（愤怒即将转化为爽感）——此时应选 shatter_strike
- [EX-02] 场景中的"愤怒"是角色之间的对等冲突（双方各有道理）——此时不应引导观众单向宣泄
- [EX-03] 场景的情绪主导是"悲伤"而非"愤怒"（被欺压后的委屈）——此时应选 tear_resonance

**P4 选择优先级**：
- anger_release > shatter_strike：当情绪还在积累阶段
- anger_release > tear_resonance：当"愤怒"多于"悲伤"
- team_cheer > anger_release：当场景有两方对等对抗且观众需要表态时

**P5 正例与反例**：
- ✅ 正例：皇帝看到权贵子弟全是废物，气得浑身发抖——观众共情皇帝的愤怒和失望
- ✅ 正例：蛮夷当街强抢民女——观众对暴行产生道德愤怒
- ❌ 反例：蛮夷被高手打脸——此时应该释放（shatter_strike），而非继续愤怒
- ❌ 反例：主角因自身错误导致失败——愤怒缺少清晰的道德指向

---

### 2.3 laugh_burst（大笑互动）

**P1 核心语义定义**：
场景本身具有**明确的喜剧意图**，观众会产生自发性的笑。核心心理动作是"哈哈"——不是微笑、不是觉得有趣、不是荒诞感，而是**有出声笑的冲动**。

**P2 适用场景正例**：
- (a) 台词驱动的笑点：角色说出有故意设计感的幽默对白
- (b) 行为驱动的笑点：角色做出滑稽/笨拙/夸张的肢体动作
- (c) 反差驱动的笑点：严肃场景中突然插入不协调的喜剧元素（但必须是创作者**有意设计**的喜剧时刻）
- 通用判断标准：**如果一个不带任何预设的普通观众看到这个场景，会不会笑出声？如果答案不是明确的"会"，就不选 laugh_burst**

**P3 硬排除规则**：
- [EX-01] 场景的"反差"是荒诞/离谱而非幽默——观众的心理是"这人怎么这样"（认知判断）而非"哈哈哈"（情绪释放）。**此时选 emotion_buffer 或不做互动点**
- [EX-02] 场景是角色在笑但观众不觉得好笑——角色的笑声不是判断依据
- [EX-03] 场景包含幽默元素但同时承载重要剧情信息——幽默是包装，核心是信息传递，此时不适用 laugh_burst
- [EX-04] 场景的情绪主导是"讽刺"或"黑色幽默"——此时观众的情绪复杂，不适合单一的大笑互动

**P4 选择优先级**：
- laugh_burst > emotion_buffer：当场景的喜剧意图明确且无其他叙事功能
- emotion_buffer > laugh_burst：当场景的"好笑"判断存疑时，宁可降级

**P5 正例与反例**：
- ✅ 正例：角色一本正经地做出完全错误的推理，引起观众哄笑——明确的喜剧设计
- ✅ 正例：两个角色误会对方意图而产生滑稽的对话交锋
- ❌ 反例：纨绔撒金女子哄抢——这是"离谱"和"荒诞"，观众的反应是"这人真离谱"，不是"真好笑"。**此场景的叙事功能是建立角色人设，不是制造笑点。** 不应选 laugh_burst
- ❌ 反例：反派被击败后狼狈逃窜——虽然有解气感，但不是喜剧，应选 shatter_strike

### 2.4 tear_resonance（泪点共鸣）

**P1 核心语义定义**：
场景触发了观众的**共情性悲伤或感动**。核心心理动作是"好心疼/好感人"——观众的情感与角色的遭遇产生深度共鸣。

**P2 适用场景正例**：
- (a) 离别/牺牲：角色被迫分离或做出利他性牺牲
- (b) 委屈揭露：角色的隐忍和痛苦在此刻被揭示或被人理解
- (c) 情感爆发：角色长期压抑的情感在此刻释放（哭泣/告白/忏悔）
- 通用判断标准：**此场景是否会让一个共情能力正常的观众感到"心里一酸"？**

**P3 硬排除规则**：
- [EX-01] 场景的悲伤来自反派遭受报应——观众的心理是"活该"而非"心疼"，应选 shatter_strike
- [EX-02] 场景的悲伤伴随强烈的愤怒——如果愤怒是主导情绪，应选 anger_release
- [EX-03] 角色死亡瞬间——给观众消化时间，互动点应偏移到死亡前或死亡后的情感反应段落

**P4 选择优先级**：
- tear_resonance > anger_release：当悲伤和愤怒并存，且悲伤为主导
- tear_resonance > emotion_buffer：当场景有明确的情感触动，不止是过渡

**P5 正例与反例**：
- ✅ 正例：角色A被继母毒杀生母的真相揭露，观众共情其十五年隐忍
- ✅ 正例：一直坚强的角色终于在信任的人面前崩溃流泪
- ❌ 反例：反派的悲惨过去被揭示——虽然可悲，但观众的情感是复杂的（恨意 + 零星同情），不适合单一泪点
- ❌ 反例：角色被轻微感动——情感强度不够，应选 emotion_buffer

### 2.5 team_cheer（站队助威）

**P1 核心语义定义**：
场景中两方形成**道义上大致对等的对抗**，观众需要选择支持哪一方来表达自己的价值观。核心心理动作是"我支持这一边"——选择本身有意义，因为两边都有合理之处。

**关键约束：此组件要求对抗双方的道义立场大致对等。这不是"好人 vs 坏人"的站队，而是"两种价值观/两种选择"的表态。**

**P2 适用场景正例**：
- (a) 价值观对抗：两个角色代表了不同但都有合理性的立场（如"坚持原则 vs 灵活变通"）
- (b) 阵营选择：观众需要在两个有各自魅力的阵营中做选择（如"支持太子党还是皇后党"）
- (c) 角色对决：两个都有粉丝基础的角色正面对决，观众的情感倾向分化
- 通用判断标准：**是否存在至少一部分观众会真诚地选择"对方"那一侧？如果 90%+ 的观众都会选同一侧，则此场景不适用 team_cheer**

**P3 硬排除规则**：
- [EX-01] **对抗双方存在明确的道义落差**：一方是明显的"正义"、另一方是明显的"邪恶"——此时选 team_cheer 是将不道德的一方放在"可以被选择"的位置上，构成对受害方的二次冒犯。**此时应选 anger_release 或 guardian_shield。**
- [EX-02] 场景中只有一方在行动（另一方是被动的承受者）——不存在对等的"对抗"关系
- [EX-03] 场景的本质是"外敌入侵"或"民族冲突"——此类场景中，观众的身份认同已经天然决定了立场，"选择"是虚假的

**P4 选择优先级**：
- anger_release > team_cheer：当对抗双方有道义落差（一方明显为恶）
- guardian_shield > team_cheer：当观众需要的是"守护"某一方而非"选择"
- team_cheer > emotion_buffer：当确实存在道义对等的两方对峙

**P5 正例与反例**：
- ✅ 正例：朝堂上，主战派（捍卫尊严）vs 主和派（避免牺牲）——双方各有合理考量
- ✅ 正例：爱情剧中，两个追求者各有优点，女主角面临选择——观众可以真诚地各自站队
- ❌ 反例：**蛮夷（侵略者/施暴者）vs 示例王朝百姓（受害者/被侵略方）——这是施暴者与受害者的关系，不是"两种价值观"的关系。让观众选择"站队蛮夷"等同于让观众认同侵略行为。** 应选 anger_release
- ❌ 反例：抗日剧中问观众"站中国还是日本"——同理，应选 anger_release 或 guardian_shield

### 2.6 guardian_shield（守护加持）

**P1 核心语义定义**：
当角色处于危险/弱势/受伤害的状态时，观众产生**保护欲和支持冲动**。核心心理动作是"不要伤害他/她"——观众想为角色提供一种情感上的"护盾"。

**P2 适用场景正例**：
- (a) 角色受伤或处于危险：观众担心角色的安全
- (b) 角色处于逆风/劣势：观众想给角色"加油"
- (c) 角色的脆弱面暴露：强大角色展示脆弱时刻，激发观众的保护欲
- 通用判断标准：**观众在此刻的直觉反应是"不要/小心/撑住"，而非"打回去"（那是 anger_release）或"选哪边"（那是 team_cheer）**

**P3 硬排除规则**：
- [EX-01] 角色处于劣势但即将反击——此时观众的心理是期待（prediction_card）或蓄力（shatter_strike），而非守护
- [EX-02] 守护对象是反派——即使反派有悲惨过去，守护反派的"安全"与之前的叙事立场矛盾
- [EX-03] 场景的核心情绪是愤怒而非担忧——此时应选 anger_release

**P4 选择优先级**：
- guardian_shield > anger_release：当观众心理是"保护"而非"反击"
- team_cheer > guardian_shield：当场景有两方对等对抗时

**P5 正例与反例**：
- ✅ 正例：示例王朝无人可用，满朝焦虑，观众可以为示例王朝国运做情感守护
- ✅ 正例：角色孤身一人面对压倒性敌人，观众想给他力量
- ❌ 反例：角色已经占据上风——此时不适配守护
- ❌ 反例：守护反派不被正义方击败——与叙事立场矛盾

### 2.7 prediction_card（剧情预测）

**P1 核心语义定义**：
场景到达了一个**叙事分叉点**——接下来有多种可能的走向，观众需要判断剧情的走向。核心心理动作是"接下来会怎样？"——这是**前瞻性**的悬念。

**P2 适用场景正例**：
- (a) 反转前夕：信息已经给足，观众可以做出有依据的预判
- (b) 角色面临选择：角色需要在多个选项中做决定，观众可以猜测其选择
- (c) 悬念升起：一个谜题被提出但答案尚未揭晓
- 通用判断标准：**此刻观众是否能基于已有信息做出有意义的预判？如果答案还没揭晓但线索已给足，适合 prediction_card**

**P3 硬排除规则**：
- [EX-01] 场景的悬念是"这东西意味着什么"（回溯推理）而非"接下来会发生什么"（前瞻预测）——此时应选 clue_judge_card
- [EX-02] 答案已经显而易见到不需要预判——没有真正的悬念
- [EX-03] 预判的答案无法在后续剧集中得到验证——违反"预测可验证"原则

**P4 选择优先级**：
- clue_judge_card > prediction_card：当悬念的本质是"推理"而非"预判"
- prediction_card > emotion_buffer：当有清晰的叙事分叉点

**P5 正例与反例**：
- ✅ 正例：皇帝下令测试权贵子弟——观众可以预判测试结果（谁强谁弱）
- ✅ 正例：角色收到两封密信，必须选择先打开哪一封——明确的叙事分支
- ❌ 反例：玉佩的出现让人好奇它的来历——这是"这个道具代表什么"（回溯推理），不是"接下来会怎样"（前瞻预测），应选 clue_judge_card
- ❌ 反例："主角会不会死"——在非结局集中，主角死亡概率为 0，这不是有意义的预测

---

### 2.8 clue_judge_card（线索判断）

**P1 核心语义定义**：
场景中出现了一个需要观众**推理和判断**的关键信息——这个信息是伏笔、证据或暗示，观众需要自己理解它的含义。核心心理动作是"这意味着什么？"——这是**回溯性**的推理。

**P2 适用场景正例**：
- (a) 伏笔埋设：一个不起眼的细节暗示了后续剧情
- (b) 证据出现：关键物品/证词浮出水面，观众需要判断其意义
- (c) 谎言与真相：角色说出的话与观众已知信息矛盾，需要判断谁在说谎
- 通用判断标准：**此刻的叙事中是否有一个"需要观众主动思考才能理解"的信息点？**

**P3 硬排除规则**：
- [EX-01] 线索的意义已经被角色当场解释清楚——观众不需要自己推理，不适合 clue_judge_card
- [EX-02] 线索是纯粹的"接下来会怎样"（前瞻预测）——此时应选 prediction_card
- [EX-03] 线索过于隐晦，普通观众几乎不可能注意到——互动点建立在一个大多数观众不会注意的细节上，效果差

**P4 选择优先级**：
- clue_judge_card > prediction_card：当悬念的本质是"推理"而非"预判"
- emotion_buffer > clue_judge_card：当线索太弱，不值得单独设互动点

**P5 正例与反例**：
- ✅ 正例：角色A护住玉佩——"平日挥金如土的纨绔为什么唯独死守这块玉佩？"观众需要推理
- ✅ 正例：两个角色对同一事件给出矛盾的说法——谁在说谎？
- ❌ 反例：角色直接说出"这块玉佩是我母亲留给我的"——线索已被解释，不需要观众推理
- ❌ 反例：一个极其模糊的背景细节——观众几乎不可能注意到

---

### 2.9 celebrate_confetti（庆祝礼炮）

**P1 核心语义定义**：
角色达成了**明确的正向目标**，观众和角色一起庆祝。核心心理动作是"太好了！"——与 shatter_strike 不同，这里没有"击败对手"的攻击性，只有"达成目标"的喜悦。

**P2 适用场景正例**：
- (a) 目标达成：角色完成了一个长期努力的目标
- (b) 团圆/重聚：分离的角色重逢
- (c) 胜利时刻（非对抗性）：比武胜出、考试通过、事业成功
- 通用判断标准：**情绪的核心是"喜庆"而非"解气"**

**P3 硬排除规则**：
- [EX-01] 喜悦来自击败/羞辱对手——此时应选 shatter_strike
- [EX-02] 胜利伴随沉重代价——喜悦不纯粹，不适合庆祝
- [EX-03] 场景是过渡性的小胜利——情绪强度不够，应选 emotion_buffer

**P4 选择优先级**：
- shatter_strike > celebrate_confetti：当喜悦与"击败对手"强相关
- celebrate_confetti > emotion_buffer：当情绪强度足够且正向纯粹

**P5 正例与反例**：
- ✅ 正例：比武胜利后，角色获得众人认可和荣誉——纯粹的成就庆祝
- ✅ 正例：分离多年的亲人终于团聚——情感正向且不涉及对手
- ❌ 反例：打脸反派后庆祝——核心是"解气"而非"喜庆"，应选 shatter_strike
- ❌ 反例：角色勉强险胜且付出重大代价——情绪复杂，不适用

---

### 2.10 sugar_storm（撒糖风暴）

**P1 核心语义定义**：
场景中的角色关系（通常是CP/情侣）出现**甜蜜升温**，观众产生"嗑到了"的满足感。核心心理动作是"好甜！"——观众对角色关系的情感投入获得了正向回报。

**P2 适用场景正例**：
- (a) CP 互动高甜时刻：表白、牵手、拥抱、暧昧对话
- (b) 角色关系突破：从敌对到友好、从疏远到亲近
- (c) 保护/体贴举动：一个角色默默为另一个角色付出
- 通用判断标准：**观众看到这个场景是否会不由自主地露出"姨母笑"？**

**P3 硬排除规则**：
- [EX-01] 甜蜜是虚假的（一方在欺骗另一方）——观众的知情破坏了甜度
- [EX-02] 场景的核心是"搞笑"而非"甜蜜"——应选 laugh_burst
- [EX-03] 非 CP 关系（如母女情深）——"甜蜜"在此语境中仅适用于浪漫关系或准浪漫关系

**P4 选择优先级**：
- sugar_storm > laugh_burst：当甜蜜和搞笑并存，甜蜜为主导
- sugar_storm > emotion_buffer：当关系升温值得标记

**P5 正例与反例**：
- ✅ 正例：男主为女主披上外衣，两人四目相对——经典甜宠时刻
- ✅ 正例：傲娇角色终于说出真心话——关系突破
- ❌ 反例：两人吵架后的和解——和解的情感复杂（歉意/释然），不完全是"甜"
- ❌ 反例：一方在算计另一方但假装甜蜜——观众知情，不适配

---

### 2.11 emotion_buffer（情绪缓冲）

**P1 核心语义定义**：
在两个高强度情绪段落之间，提供一个**情感过渡和喘息空间**。也作为"兜底"——当一个场景有值得标记的叙事信息但情绪强度不足以匹配任何特定组件时使用。

核心心理动作是"注意这里有变化/过渡"——不是释放情绪，而是**标记一个叙事呼吸点**。

**P2 适用场景正例**：
- (a) 高强度段落后：激烈冲突/高潮后的短暂平静
- (b) 叙事信息传递：关键信息被传递但场景本身不带强情绪
- (c) 角色建立时刻：重要角色的首次出场或关键特征展示（非情绪高潮）
- (d) 任何有叙事价值但不属于其他 11 种组件范畴的场景
- 通用判断标准：**这个场景有被标记的价值，但没有强到触发任何特定情绪组件——它就是合适的 emotion_buffer 场景**

**P3 硬排除规则**：
- [EX-01] 场景有明确且单一的情绪指向——如果有更精确的组件适用，不用 emotion_buffer
- [EX-02] 场景完全平淡无任何叙事价值——不设互动点比用 emotion_buffer 更好
- [EX-03] 每集使用 emotion_buffer 不超过 2 次——它是兜底，不是主力

**P4 选择优先级**：
- 所有其他 11 个组件 > emotion_buffer：buffer 永远是最后选择
- 不设点 > emotion_buffer：当场景同时也没有叙事价值时

**P5 正例与反例**：
- ✅ 正例：紧张朝堂戏结束后，画面切换到安静的庭院过渡镜头——给观众呼吸空间
- ✅ 正例：皇帝得知比武招亲困局——建立核心冲突，情绪尚未到达愤怒的强度
- ❌ 反例：反派被打脸——有明确的 shatter_strike 可用，不应降级到 buffer
- ❌ 反例：一连串无意义的过渡画面——不设点

---

### 2.12 episode_end_prediction（剧尾预测）

**P1 核心语义定义**：
在集尾悬念处，观众对下一集（或更远集数）的剧情做出预判。是 prediction_card 在集尾位置的特殊形态，但只在**集尾 ±12 秒**范围内使用。

**P2 适用场景正例**：
- (a) 集尾 cliffhanger 后：悬念被抛出，观众自然想知道后续
- (b) 未解决的冲突：角色面临未完成的挑战
- 通用判断标准：**此集结尾有没有一个让观众"必须看下一集"的钩子？**

**P3 硬排除规则**：
- [EX-01] 不在集尾 ±12 秒范围内 — 使用 prediction_card
- [EX-02] 预测的答案无法在后续集中验证
- [EX-03] 预测题是纯粹的"猜谜"（无任何线索支撑）

**P4 选择优先级**：
- episode_end_prediction > prediction_card：仅在集尾时间范围内

**P5 正例与反例**：
- ✅ 正例：神秘高手秒杀蛮夷后，镜头切到角色A——"出手的人是谁？"
- ✅ 正例：角色打开一封未读的密信，集终——"信里写了什么？"
- ❌ 反例：集中间段的悬念——应用 prediction_card
- ❌ 反例："下一集会不会有新人登场？"——纯猜测无依据

## 三、组件选择决策树

LLM 在选择组件时按以下流程执行：

```
Step 0: 场景叙事功能分类
  ├─ 这是情绪释放时刻吗？
  │   ├─ 是，释放的核心情绪是什么？
  │   │   ├─ 解气/爽感 → shatter_strike（检查 EX-01/02/03）
  │   │   ├─ 愤怒/不公 → anger_release（检查 EX-01/02/03）
  │   │   ├─ 悲伤/感动 → tear_resonance（检查 EX-01/02/03）
  │   │   └─ 好笑 → laugh_burst（检查 EX-01/02/03/04）
  │   └─ 否 → 继续
  │
  ├─ 这是需要观众表态的对抗吗？
  │   ├─ 是，双方道义对等吗？
  │   │   ├─ 是 → team_cheer（检查 EX-01/02/03）
  │   │   └─ 否 → guardian_shield 或 anger_release
  │   └─ 否 → 继续
  │
  ├─ 这是需要观众思考的悬念吗？
  │   ├─ 是，"接下来会怎样"（前瞻预测）？→ prediction_card
  │   ├─ 是，"这意味着什么"（回溯推理）？→ clue_judge_card
  │   └─ 否 → 继续
  │
  ├─ 这是正向庆祝/甜蜜吗？
  │   ├─ 击败对手后的喜悦 → shatter_strike
  │   ├─ 目标达成的纯粹喜悦 → celebrate_confetti
  │   └─ CP 甜蜜升温 → sugar_storm
  │
  ├─ 这是集尾悬念吗？→ episode_end_prediction
  │
  └─ 以上都不是但有叙事价值 → emotion_buffer
      └─ 以上都不是且无叙事价值 → 不设互动点
```

## 四、关键语义区分表

以下是容易被 LLM 混淆的组件对，附精确区分标准：

| 易混淆对 | 区分标准 |
|---------|---------|
| **shatter_strike vs anger_release** | 情绪是积累阶段(anger)还是释放阶段(shatter)？看此场景之前是否有同向情绪积累 |
| **shatter_strike vs celebrate_confetti** | 喜悦来自"击败对手"(shatter)还是"达成目标"(celebrate)？ |
| **laugh_burst vs emotion_buffer** | 场景有**明确喜剧意图**(laugh)还是只是**荒诞/离谱**(buffer)？问：普通观众会不会笑出声？ |
| **team_cheer vs anger_release** | 对抗双方**道义对等**(team)还是**有道义落差**(anger)？问：至少有一部分观众会真诚选对方吗？ |
| **team_cheer vs guardian_shield** | 观众需要**选择**(team)还是**守护**(guardian)？ |
| **prediction_card vs clue_judge_card** | 悬念是**前瞻**("接下来")还是**回溯**("意味着什么")？ |
| **tear_resonance vs anger_release** | 被欺压后的情绪是**委屈/心疼**(tear)还是**愤怒/不公**(anger)？ |

## 五、应用示例：用精确标准重新评估 IP2 和 IP5

### IP5（蛮夷辱国）用新标准决策：

```
Step 0: 场景功能分类
  → 观众产生道德愤怒，是情绪释放时刻

情绪释放的核心是什么？
  → 愤怒/不公 → 考虑 anger_release

检查 anger_release EX 规则：
  EX-01(已到释放点？): 蛮夷还在嚣张，反转未到 → 不是释放点 → anger_release 适用 ✅
  EX-02(对等冲突？): 一方是施暴者一方是受害者 → 不是对等冲突 ✅
  EX-03(悲伤主导？): 愤怒是多于悲伤的 ✅

是否考虑 team_cheer？
  → 检查 team_cheer EX-01(道义落差):
    蛮夷=侵略者/施暴者, 示例王朝百姓=受害者
    → 明确道义落差 → team_cheer 不可用 ❌

结论: anger_release ✅  team_cheer ❌
```

### IP2（纨绔撒金）用新标准决策：

```
Step 0: 场景功能分类
  → 场景是展示角色A的"纨绔人设"，不是情绪释放

这是需要观众思考的悬念吗？→ 否
这是需要观众表态的对抗吗？→ 否
这是正向庆祝/甜蜜吗？→ 否

考虑 laugh_burst？
  → 检查 laugh_burst EX-01(荒诞vs幽默):
    撒金女子哄抢 → 观众反应是"这人真离谱"(认知判断)
    不是"真好笑"(情绪释放)
    → laugh_burst 不可用 ❌

考虑 emotion_buffer？
  → 场景有叙事价值(建立角色A人设) → emotion_buffer 适用 ✅

结论: emotion_buffer ✅  laugh_burst ❌
```

## 六、组件库维护方需确认的事项

1. 每个组件的 P3 硬排除规则是否需要增删？（当前基于第一轮运行中的错误案例推导，可能不完整）

2. §1.3 的场景叙事功能分类体系是否被认可？如果不认可，LLM 的"先分类再选择"流程需要对应的调整。

3. emotion_buffer 作为"兜底组件"的定位是否被接受？当前设计中近 50% 的候选点可能最终落入 emotion_buffer。

4. team_cheer 的"道义对等"约束是最关键的修正——确认前端交互设计是否接受"道义不对称场景禁用站队组件"这一原则。
