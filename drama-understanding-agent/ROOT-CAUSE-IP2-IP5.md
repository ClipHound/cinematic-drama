> 日期: 2026-06-10

> 对应修复方案: `SPEC-interaction-component-standard.md`（标准制定）→ `INTERACTION-COMPONENT-GUIDE.md`（组件库落地）

## 一、IP5：蛮夷辱国场景 → team_cheer（站队助威）

### 1.1 最终产出的错误表现

```json
{
  "id": "ip_ep_001_0005",
  "start_ms": 243000, "end_ms": 252000,
  "component": "team_cheer",
  "emotion": "support",
  "score_type": "cocreate",
  "config": {
    "sides": [
      {"label": "示例王朝子民", "character": "全城百姓"},
      {"label": "嚣张蛮夷", "character": "蛮夷壮汉"}
    ]
  }
}
```

组件 `team_cheer` 在此场景的含义是：让观众在"示例王朝子民"和"嚣张蛮夷"之间选择站队。问题本质：**将施暴者（蛮夷）放在了与受害者（示例王朝百姓）对等的道义位置上，邀请观众"选择"是否认同侵略行为。**

### 1.2 传播链路逐环节追踪

**环节 1：视频实际内容（Ground Truth）**

ASR 记录的场景台词：
```
[03:47] 这些蛮人光天化日不仅强抢民女还敢打人
[04:03] 他们能奈我何别说他们就连你们的皇帝老儿来了
        所以我们都在卑躬屈膝
```

画面：蛮夷壮汉当街强抢民女、对围观百姓吐唾沫、口出狂言羞辱示例王朝皇帝。"蛮夷"在古装剧语义场中天然携带"外敌/侵略者/反派"标签。场景的戏剧功能是**制造道德愤怒**——施暴者 vs 受害者，正邪分明。

---

**环节 2：VLM（Agent 1）的视频理解——正确**

VLM 产出的候选点：

```json
{
  "start_ms": 243000, "end_ms": 253000,
  "anchor_line": "就连你们的皇帝老儿来了，对我们都得毕恭毕敬",
  "emotion_type": "tense+angry",
  "intensity": 0.9,
  "reason": "蛮夷嚣张到当众羞辱示例王朝最高统治者，把观众的愤怒情绪直接推到顶点，
             所有人都等着看打脸反转",
  "visual_cue": "留着长辫戴兽骨项链的蛮夷壮汉满脸嚣张，对着围观百姓吐唾沫"
}
```

VLM 的判断均正确：
- 核心情绪标注为 `tense+angry`——不是 support、不是 curious
- reason 正确识别了场景功能——"愤怒情绪推到顶点"
- visual_cue 正确捕捉了蛮夷的负面形象细节（"吐唾沫"）
- 未出现任何暗示"站队"或"选择"的关键词

**VLM 没有错。** 它正确理解了这是一个激发愤怒的不对称对抗场景。

---

**环节 3：Agent 2（Interaction LLM）的输入与推理——形式正确但语义缺失**

Agent 2 接收到的关键信号：
- VLM 候选点的 `emotion_type: "tense+angry"`
- 组件库中 team_cheer 的描述：

```
team_cheer: 站队助威。适合对峙、阵营选择、角色对抗。emotion=support
```

Agent 2 的推理链（反推）：
1. 场景中有两个明确对立方（蛮夷 vs 示例王朝百姓）
2. 组件库说 team_cheer 适合"对峙、阵营选择、角色对抗"
3. 三个触发词全部命中
4. 因此选择 team_cheer 并填充 config.sides

**Agent 2 的形式推理是正确的**（对峙场景→站队组件），但它缺少一个关键的语义判断——**这个对峙双方在道义上是否对等？**

---

**环节 4：组件库描述——缺失关键约束**

旧组件库中 team_cheer 的描述：

```
team_cheer: 站队助威。适合对峙、阵营选择、角色对抗。emotion=support
```

只声明了 P2（何时选），缺少 P3（何时不能选）。具体缺失：

| 缺失的约束 | 为什么需要 |
|-----------|----------|
| 双方道义立场需大致对等 | 施暴者 vs 受害者关系中，"站队施暴者"不是有效选项 |
| 不适合外敌入侵/民族冲突场景 | 观众的身份认同已天然决定立场 |
| 需存在至少一部分观众会真诚选对方 | 99% 选同一侧 = 虚假选择 |

Agent 2 因为收不到任何"反例"，基于正向触发词做了它认为正确的匹配。

---

**环节 5：新组件库（INTERACTION-COMPONENT-GUIDE.md）中的修复**

team_cheer 的 P3 增加了三条硬排除规则：

```
- 好人对坏人、施暴者对受害者等存在明显道义落差时禁止使用。
- 只有一方主动行动、另一方被动承受时禁止使用。
- 外敌入侵、民族冲突等立场天然确定的场景禁止使用。
```

决策树中对应分支：
```
双方道义不对等且观众愤怒 → anger_release
```

IP5 走新规则：场景在 §3 Step 2 被识别为"双方道义不对等"→ 路由到 anger_release，team_cheer 不可达。

---

### 1.3 根因总结

| 环节 | 状态 | 问题性质 |
|------|:---:|---------|
| 视频内容 | 正邪分明的对抗 | — |
| VLM 理解 | 正确：tense+angry | — |
| Agent 2 推理 | 形式正确，但语义判断不足 | 缺少对"道义对等性"的评估 |
| 组件库描述 | 只有正向触发词，无反向排除条件 | **根因所在** |
| 最终选择 | 错误：team_cheer | anger_release 才是正确答案 |

**根因不在模型能力，在组件库的语义定义不完整。** 给组件库补上 P3 硬排除规则即可系统性解决此类问题，无需修改模型或 Prompt。

## 二、IP2：角色X撒金场景 → laugh_burst（大笑互动）

### 2.1 最终产出的错误表现

```json
{
  "id": "ip_ep_001_0002",
  "start_ms": 83000, "end_ms": 91000,
  "component": "laugh_burst",
  "emotion": "funny",
  "title": "纨绔公子当众撒金",
  "key_line": "今天全场消费都由本公子买单",
  "highlight_reason": "角色X纨绔名场面反差感拉满，用喜剧效果快速建立观众对他废物人设的初始认知"
}
```

组件 `laugh_burst` 在此场景的含义是：这是一个喜剧时刻，观众应该大笑。问题本质：**此场景的叙事功能是"建立角色人设"（认知层面的"这人真离谱"），而非"制造笑点"（情绪层面的"哈哈哈"）。场景的"反差"是结构反差（纨绔行为 vs 正常行为），不是喜剧反差（预期 vs 幽默反转）。**

### 2.2 传播链路逐环节追踪

**环节 1：视频实际内容（Ground Truth）**

ASR：
```
[01:21] 外面为何如此喧哗今天全场消费都有本公子买单
[01:31] 此子是谁竟然如此跋扈呀外地来的吧
```

画面：角色X站在高处撒银票，红毯上一群青楼女子弯腰哄抢。

此场景在短剧叙事中的功能是**建立角色X的纨绔人设**——观众需要知道"角色X是个挥霍无度的废物"，这是后续"隐藏高手"反转的铺垫。观众的心理反应是"这人怎么这样"（认知判断 + 荒诞感），不是"哈哈哈"（情绪释放）。

---

**环节 2：VLM（Agent 1）的视频理解——偏差出现**

VLM 产出的候选点：

```json
{
  "start_ms": 83000, "end_ms": 91000,
  "anchor_line": "今天全场消费都由本公子买单",
  "emotion_type": "funny+shocking",
  "intensity": 0.8,
  "reason": "顶级纨绔当众撒金的名场面，满地银票女子哄抢的画面反差感拉满，极具戏剧张力",
  "visual_cue": "红毯上散落满银票，一众穿古装裙装的女子弯腰哄抢银票"
}
```

偏差出现在情绪标签的选择上。VLM 的情绪分类体系只有 9 种标签：

```
anger / sweet / funny / sad / shocking / satisfying / tense / curious / guard
```

缺少以下精确标签：
- `absurd`（荒诞）
- `ridiculous`（离谱）
- `satirical`（讽刺）

VLM 实际想表达的是"荒诞反差"——一个人挥霍到离谱的程度让人感到荒唐——但它的词汇表里没有 `absurd`，只能用最接近的组合 `funny+shocking` 来近似：
- `shocking` 承载"离谱/荒唐"的维度
- `funny` 承载"反差"的维度

**VLM 的偏差不是"看错了画面"（visual_cue 是准确的），而是"情绪标签体系的粒度不够"**。它被限制在 9 个标签内做近似表达，导致下游 Agent 2 收到了有噪声的情绪信号。

---

**环节 3：Agent 2（Interaction LLM）的输入与推理——关键词匹配但语义漂移**

Agent 2 接收到的关键信号：
- VLM 候选点的 `emotion_type: "funny+shocking"`
- reason 中的关键词："反差感拉满"

Agent 2 查看组件库中 emotion=funny 的组件：

```
laugh_burst: 大笑互动。适合反差笑点、尴尬喜剧、爽快吐槽。emotion=funny
```

触发关键词匹配：
- `funny` → 组件 emotion 标签匹配
- "反差感拉满"（VLM reason）→ "反差笑点"（组件描述）→ **"反差"一词的语义漂移发生**

**"反差"在两个系统中的含义不同**：

| 使用方 | 文本 | 实际含义 |
|--------|------|---------|
| VLM reason | "反差感拉满" | 角色X的行为与正常人行为的**结构反差**（离谱 vs 正常） |
| 组件库描述 | "反差笑点" | 预期与结果之间的**喜剧反差**（以为会 A 结果却是 B） |
| Agent 2 的理解 | 将两者等量齐观 | 认为 VLM 说的"反差"就是组件库的"反差笑点" |

Agent 2 完成了关键词对齐，但两个"反差"指向不同的语义。这不是 Agent 2 "不理解"，而是**同一个词在两个系统中的指涉不同**——Agent 2 没有上下文来区分这种差异。

---

**环节 4：旧组件库描述——"反差笑点"的边界未定义**

旧 laugh_burst 描述：

```
laugh_burst: 大笑互动。适合反差笑点、尴尬喜剧、爽快吐槽。emotion=funny
```

缺失的约束：
- 未定义什么是"笑点"——荒诞≠好笑
- 未区分"结构反差"和"喜剧反差"
- 未给出判断标准——谁来判断"好不好笑"？

---

**环节 5：新组件库（INTERACTION-COMPONENT-GUIDE.md）中的修复**

laugh_burst 的 P3 增加了四条硬排除规则，第一条直接命中：

```
- 只有荒诞、离谱或尴尬感，但没有明确喜剧意图时不得使用。
```

P2 判断标准收紧：

```
判断标准：不了解上下文的普通观众也能明确识别这是一个笑点。
```

P5 反例直接列入本场景：

```
- 反例：纨绔撒钱引发哄抢，主要作用是建立离谱人设。
```

IP2 走新规则：检查 laugh_burst → P3 EX-01 触发（"荒诞、离谱但无喜剧意图"）→ laugh_burst 不可用 → 降级到 emotion_buffer 或不设点。

---

### 2.3 根因总结

IP2 的错误是两个环节的叠加效应：

| 环节 | 状态 | 问题性质 |
|------|:---:|---------|
| 视频内容 | 建立角色人设的荒诞场景 | — |
| VLM 理解 | 偏差：情绪标签粒度过粗 | 9 种标签缺少 "absurd" 类，只能以 funny+shocking 近似 |
| Agent 2 推理 | 关键词匹配但语义漂移 | VLM 的"反差"（结构反差）≠ 组件库的"反差"（喜剧反差） |
| 组件库描述 | 未定义 laugh_burst 的适用范围边界 | 与 IP5 同根因 |
| 最终选择 | 错误：laugh_burst | emotion_buffer 更合适 |

**第一根因**：VLM 情绪标签体系粗粒度（9 种不够）→ 上游信号有噪声。
**第二根因**：组件库缺少 P3 排除规则和精确判断标准 → 下游无法过滤噪声。
**两个根因叠加**：上游信号偏差 + 下游无过滤 = 错误选择。

新组件库通过 P3 硬排除规则（"荒诞但无喜剧意图→不用"）在第二根因处切断了错误传播链。第一根因（情绪标签粒度）的解决需要扩展 VLM 的情绪分类体系，不在组件库范围内。

## 三、两个错误的共同模式

### 3.1 传播链路的同构性

```
IP5: 视频 → VLM(✅正确) → Agent2(⚠️缺道义判断) → 组件库(❌无排除规则) → 错误
IP2: 视频 → VLM(⚠️标签粒度) → Agent2(⚠️语义漂移) → 组件库(❌无排除规则) → 错误
                                                         ↑
                                                  共同根因所在
```

### 3.2 三个系统性因素

**因素 1：VLM 情绪标签体系粗粒度（影响 IP2）**

9 种情绪标签无法覆盖短剧中的精细情感。`absurd`（荒诞）、`ridiculous`（离谱）、`indignant`（义愤）等关键标签缺失，导致 VLM 只能用近似标签表达复杂情感，向上游 Agent 2 传递有噪声的信号。

**因素 2：组件库只有正向触发词，无反向排除条件（影响 IP2 和 IP5）**

旧组件库每个组件只有一行描述，格式为"组件名: 一句话定义。适合 A、B、C"。当场景命中 A/B/C 中的任意关键词时，Agent 2 就会选择该组件。但组件库没有声明"不适合 X、Y、Z"——Agent 2 收不到任何反例信号。

这是两个错误的**共同根因**。新组件库通过 P3 硬排除规则解决了此问题。

**因素 3：Agent 2 对 VLM 输出是"信任模式"（影响 IP2 和 IP5）**

Agent 2 消费 VLM 的情绪标签时，不质疑、不校准。当 VLM 的情绪标签存在偏差（IP2）时，Agent 2 忠实地将偏差传导到最终选择中。串行流水线架构中，下游信任上游，错误在上游产生后不会被拦截。

新组件库的 P3 硬排除规则部分缓解了此问题——即使上游信号有噪声，下游的排除规则也能在最后一环拦截。

### 3.3 新组件库修复覆盖情况

| 错误 | 修复手段 | 修复位置 |
|------|---------|---------|
| IP5: team_cheer 用于道义不对称场景 | P3 硬排除：好人对坏人/施暴者对受害者禁止 | team_cheer §2.5 |
| IP5: team_cheer 用于外敌入侵场景 | P3 硬排除：外敌入侵/民族冲突禁止 | team_cheer §2.5 |
| IP2: laugh_burst 用于荒诞但非喜剧场景 | P3 硬排除：荒诞/离谱但无喜剧意图禁止 | laugh_burst §2.4 |
| IP2: "反差"语义漂移 | P2 判断标准收紧：普通观众能否明确识别为笑点 | laugh_burst §2.4 |
| IP2 场景本身 | P5 反例直接列入"纨绔撒钱" | laugh_burst §2.4 P5 |
