# 短剧互动特效组件使用规范

> 版本：1.0  
> 适用项目：Cinematic Short-Drama App  
> 对齐标准：`SPEC-interaction-component-standard.md` 2.0  
> 组件数量：12

## 1. 使用原则

互动点不能只根据 `emotion` 或台词关键词选择组件。选择时必须按以下顺序判断：

1. 判断场景的叙事功能。
2. 判断观众此刻最主要的心理动作。
3. 检查组件的硬排除规则。
4. 多个组件都适用时，按选择优先级决策。
5. 没有明确叙事价值时，不设置互动点。

### 1.1 叙事功能分类

| 叙事功能 | 组件 |
|---|---|
| 情绪宣泄 | `shatter_strike`、`anger_release`、`tear_resonance`、`laugh_burst` |
| 立场表达 | `team_cheer`、`guardian_shield` |
| 认知参与 | `prediction_card`、`clue_judge_card` |
| 氛围渲染 | `celebrate_confetti`、`sugar_storm` |
| 节奏过渡 | `emotion_buffer` |
| 集尾结构 | `episode_end_prediction` |

### 1.2 通用呈现规则

- 特效只在剧情语义成立的时间窗口内出现。
- 特效层高于视频普通文字和侧边操作按钮。
- 播放进度条、底部导航和模态弹窗层级高于普通特效。
- 特效画面、按钮、点击区域和长按区域必须使用同一定位容器，移动特效时命中区域必须同步移动。
- 用户操作特效时，不触发视频单击暂停。
- 同一时间只显示一个主要互动组件；高优先级组件可以替换低优先级组件。
- 组件结束、超时、跳集或切换视频时必须清理动画、计时器和交互层。

## 2. 组件精确定义

## 2.1 `shatter_strike` 碎屏暴击

![碎屏暴击](effect-screenshots/05-shatter-strike.png)

**P1 核心语义定义**

观众此前积累的不满、憋屈或期待在此刻获得彻底释放。核心心理动作是“终于解气了”。

**P2 适用场景正例**

- 反派在持续嚣张后被当众打脸。
- 主角长期隐忍后成功反击。
- 关键证据或实力揭晓，彻底扭转局面。
- 判断标准：此前至少有一个场景积累同向情绪，而当前场景是明确释放点。

**P3 硬排除规则**

- 当前仍在铺垫或积累愤怒时不得使用。
- 爽感伴随明显悲剧、牺牲或道德困境时不得使用。
- 主导情绪是喜剧而非解气时不得使用。

**P4 选择优先级**

- 击败对手带来的喜悦：优先于 `celebrate_confetti`。
- 愤怒仍在积累：改用 `anger_release`。
- 悲伤或心疼占主导：改用 `tear_resonance`。

**P5 正例与反例**

- 正例：主角被诬陷多集后，当众拿出证据完成翻盘。
- 正例：欺压百姓的反派被隐藏高手一击击败。
- 反例：反派刚开始嚣张，此时应使用 `anger_release` 或不设点。
- 反例：主角复仇成功却付出亲人死亡的代价，情绪不够纯粹。

---

## 2.2 `anger_release` 生气宣泄

![生气宣泄](effect-screenshots/02-anger-release.png)

**P1 核心语义定义**

观众对不公、欺压或背叛产生共情性愤怒，需要表达“太过分了”。

**P2 适用场景正例**

- 角色遭受冤枉、羞辱、欺压或背叛。
- 反派正在实施具有明确道德问题的行为。
- 角色长期处于不公平的弱势状态。
- 判断标准：愤怒仍在积累，观众需要出口，但报应或反击尚未落地。

**P3 硬排除规则**

- 反击已经成功、情绪进入爽感释放时不得使用。
- 双方立场大致对等、各有合理性时不得引导单向愤怒。
- 委屈、心疼或悲伤明显强于愤怒时不得使用。

**P4 选择优先级**

- 情绪积累阶段：优先于 `shatter_strike`。
- 道义对等的双方对抗：`team_cheer` 优先。
- 担忧角色安全而非愤怒：`guardian_shield` 优先。

**P5 正例与反例**

- 正例：反派当街欺负弱者，主角尚未出手。
- 正例：角色被错误证据冤枉，所有人都不相信他。
- 反例：反派已经被击败，应使用 `shatter_strike`。
- 反例：两个立场各有理由的角色争论，应考虑 `team_cheer`。

---

## 2.3 `tear_resonance` 泪点共鸣

![泪点共鸣](effect-screenshots/03-tear-resonance.png)

**P1 核心语义定义**

场景触发观众的共情性悲伤或感动，核心心理动作是“好心疼”或“好感人”。

**P2 适用场景正例**

- 角色离别、牺牲或被迫分离。
- 长期隐忍的委屈和痛苦被揭露。
- 角色终于哭泣、告白、忏悔或得到理解。
- 判断标准：正常观众看到此处会明显“心里一酸”。

**P3 硬排除规则**

- 悲伤对象是正在遭受报应的反派时不得使用。
- 愤怒明显强于悲伤时不得使用。
- 不要把互动点压在死亡发生的瞬间，应放在死亡前的告别或死亡后的情感反应段落。

**P4 选择优先级**

- 悲伤主导时优先于 `anger_release`。
- 有明确情感触动时优先于 `emotion_buffer`。

**P5 正例与反例**

- 正例：坚强角色终于在信任的人面前崩溃流泪。
- 正例：角色得知亲人多年来一直暗中保护自己。
- 反例：反派的悲惨过去被揭示，但观众仍以厌恶为主。
- 反例：轻微感动或普通和解，强度不足时使用 `emotion_buffer`。

---

## 2.4 `laugh_burst` 大笑互动

![大笑互动](effect-screenshots/04-laugh-burst.png)

**P1 核心语义定义**

场景具有明确的喜剧意图，观众会产生出声笑的冲动。核心心理动作是“哈哈”。

**P2 适用场景正例**

- 有意设计的幽默台词。
- 滑稽、笨拙或夸张的行为笑点。
- 严肃场景中有意插入的明确喜剧反差。
- 判断标准：不了解上下文的普通观众也能明确识别这是一个笑点。

**P3 硬排除规则**

- 只有荒诞、离谱或尴尬感，但没有明确喜剧意图时不得使用。
- 角色在笑但观众不会觉得好笑时不得使用。
- 幽默只是包装，场景主要用于传递重要信息时不得使用。
- 讽刺或黑色幽默导致情绪复杂时不得使用。

**P4 选择优先级**

- 喜剧意图明确：优先于 `emotion_buffer`。
- 是否好笑存在明显争议：降级为 `emotion_buffer` 或不设点。
- 甜蜜是主导情绪：`sugar_storm` 优先。

**P5 正例与反例**

- 正例：角色一本正经地作出完全错误的推理。
- 正例：两个角色因误会产生连续的滑稽对话。
- 反例：纨绔撒钱引发哄抢，主要作用是建立离谱人设。
- 反例：反派狼狈逃走，核心是解气，应使用 `shatter_strike`。

---

## 2.5 `team_cheer` 站队助威

![站队助威](effect-screenshots/08-team-cheer.png)

**P1 核心语义定义**

场景中两方道义大致对等，观众通过选择支持对象表达价值观或角色偏好。

**P2 适用场景正例**

- 坚持原则与灵活变通等两种合理价值观的对抗。
- 两个都有粉丝基础和合理动机的角色对决。
- 两个有各自魅力的阵营或感情对象竞争。
- 判断标准：确实有一部分观众会真诚选择任意一方，而不是被迫选择唯一正确答案。

**P3 硬排除规则**

- 好人对坏人、施暴者对受害者等存在明显道义落差时禁止使用。
- 只有一方主动行动、另一方被动承受时禁止使用。
- 外敌入侵、民族冲突等立场天然确定的场景禁止使用。

**P4 选择优先级**

- 道义不对等且观众愤怒：`anger_release` 优先。
- 观众需要保护弱势方：`guardian_shield` 优先。
- 双方确实对等且需要表态：`team_cheer` 优先于 `emotion_buffer`。

**P5 正例与反例**

- 正例：主战派与主和派围绕代价和尊严进行辩论。
- 正例：两个各有优点的追求者同时向女主表达心意。
- 反例：侵略者与被侵略百姓之间的冲突。
- 反例：反派围攻受伤主角，此时应使用 `guardian_shield`。

---

## 2.6 `guardian_shield` 守护加持

![守护加持](effect-screenshots/07-guardian-shield.png)

**P1 核心语义定义**

角色处于危险、弱势或受伤状态，观众产生保护欲。核心心理动作是“撑住”或“不要伤害他/她”。

**P2 适用场景正例**

- 角色受伤、被围困或面临危险。
- 角色处于明显逆风，观众希望给予支持。
- 强大角色暴露脆弱面，激发保护欲。
- 判断标准：观众第一反应是担心角色安全，而不是期待反击。

**P3 硬排除规则**

- 角色已经准备反击或即将占据上风时不得使用。
- 守护对象是明确反派时不得使用。
- 场景主导情绪是愤怒而不是担忧时不得使用。

**P4 选择优先级**

- 保护欲强于反击欲时优先于 `anger_release`。
- 两方道义对等并需要选择阵营时，`team_cheer` 优先。
- 反击已经爆发时，改用 `shatter_strike`。

**P5 正例与反例**

- 正例：角色孤身面对数量悬殊的敌人。
- 正例：角色受伤倒地，仍努力保护重要的人。
- 反例：角色已经掌握局面并准备击败敌人。
- 反例：反派即将受到正义方惩罚。

---

## 2.7 `prediction_card` 剧情预测

![剧情预测](effect-screenshots/09-prediction-card.png)

**P1 核心语义定义**

剧情到达前瞻性的分叉点，观众根据已有信息判断“接下来会怎样”。

**P2 适用场景正例**

- 反转前夕，已有足够线索支持预判。
- 角色面临多个选择，观众可以猜测其决定。
- 一个答案尚未揭晓的明确悬念刚刚升起。
- 判断标准：选项有真实不确定性，并能在后续剧情中验证。

**P3 硬排除规则**

- 问题是在判断已有线索的意义时不得使用，应使用 `clue_judge_card`。
- 答案显而易见或只有一个合理答案时不得使用。
- 预测结果无法在后续剧情验证时不得使用。

**P4 选择优先级**

- 回溯推理问题：`clue_judge_card` 优先。
- 明确的前瞻分叉：优先于 `emotion_buffer`。
- 位于集尾规定时间范围：`episode_end_prediction` 优先。

**P5 正例与反例**

- 正例：角色收到两封密信，必须选择先打开哪一封。
- 正例：测试开始前，观众预测谁会胜出。
- 反例：询问玉佩代表什么，应使用 `clue_judge_card`。
- 反例：普通剧情中询问主角会不会死，答案没有真实不确定性。

---

## 2.8 `clue_judge_card` 线索判断

![线索判断](effect-screenshots/10-clue-judge-card.png)

**P1 核心语义定义**

场景出现伏笔、证据或矛盾信息，观众需要进行回溯性推理，判断“这意味着什么”。

**P2 适用场景正例**

- 不起眼的动作或物品可能是伏笔。
- 关键证据或证词刚刚出现。
- 两个角色的说法相互矛盾，需要判断谁在说谎。
- 判断标准：信息已经出现，但意义尚未被角色直接解释。

**P3 硬排除规则**

- 线索意义已经被角色当场讲明时不得使用。
- 问题只关注下一步剧情发展时不得使用。
- 线索过于隐蔽、普通观众几乎无法注意时不得使用。

**P4 选择优先级**

- 回溯推理优先于 `prediction_card`。
- 线索强度不足时，`emotion_buffer` 或不设点优先。

**P5 正例与反例**

- 正例：挥金如土的角色却拼命保护一块旧玉佩。
- 正例：两个人对同一晚的行踪给出冲突证词。
- 反例：角色已经明确说明玉佩的来历。
- 反例：背景中一闪而过且无法辨认的物品。

---

## 2.9 `celebrate_confetti` 庆祝礼炮

![庆祝礼炮](effect-screenshots/01-celebrate-confetti.png)

**P1 核心语义定义**

角色达成明确的正向目标，观众和角色共同庆祝。核心心理动作是“太好了”。

**P2 适用场景正例**

- 长期目标终于完成。
- 分离的亲人或伙伴重新团聚。
- 考试、事业、比赛等非攻击性目标取得成功。
- 判断标准：喜悦纯粹，重点是达成目标而非羞辱或击败对手。

**P3 硬排除规则**

- 喜悦主要来自打脸、羞辱或击败对手时不得使用。
- 胜利伴随重大牺牲或沉重代价时不得使用。
- 只是普通的小进展，情绪强度不足时不得使用。

**P4 选择优先级**

- 击败对手后的喜悦：`shatter_strike` 优先。
- 纯粹目标达成：优先于 `emotion_buffer`。

**P5 正例与反例**

- 正例：角色通过努力获得认可和荣誉。
- 正例：失散多年的亲人终于相认。
- 反例：当众打脸反派后庆祝，应使用 `shatter_strike`。
- 反例：角色险胜但失去重要同伴，情绪不纯粹。

---

## 2.10 `sugar_storm` 撒糖风暴

![撒糖风暴](effect-screenshots/06-sugar-storm.png)

**P1 核心语义定义**

浪漫关系或准浪漫关系明显升温，观众产生“嗑到了”的满足感。

**P2 适用场景正例**

- 表白、牵手、拥抱或高甜暧昧对话。
- 两人从疏远、敌对转向亲近。
- 一方为另一方作出具有浪漫意味的体贴举动。
- 判断标准：观众会产生明显的甜宠满足感，而不是普通感动。

**P3 硬排除规则**

- 一方正在欺骗或利用另一方，且观众知情时不得使用。
- 场景核心是搞笑而不是甜蜜时不得使用。
- 亲情、友情等非浪漫关系不得使用。

**P4 选择优先级**

- 甜蜜与搞笑并存且甜蜜主导：优先于 `laugh_burst`。
- 关系升温明显：优先于 `emotion_buffer`。
- 保护欲而非甜蜜主导：考虑 `guardian_shield`。

**P5 正例与反例**

- 正例：男主默默给女主披上外衣，两人四目相对。
- 正例：傲娇角色终于承认真心。
- 反例：两人吵架后复杂地和解，情绪不只是甜蜜。
- 反例：一方假装深情，实际正在算计另一方。

---

## 2.11 `emotion_buffer` 情绪缓冲

![情绪缓冲](effect-screenshots/12-emotion-buffer.png)

**P1 核心语义定义**

在高强度情绪段落之间提供过渡，也用于标记有叙事价值但不适合其他精确组件的场景。

**P2 适用场景正例**

- 激烈冲突或高潮之后的短暂平静。
- 关键信息正在传递，但场景没有强烈单一情绪。
- 重要角色首次出场或关键人设建立。
- 判断标准：场景值得观众注意，但无法合理归入其他 11 个组件。

**P3 硬排除规则**

- 已有更精确的组件适用时不得使用。
- 场景完全平淡、没有叙事价值时不得使用。
- 每集最多使用 2 次，避免成为默认组件。

**P4 选择优先级**

- 其他所有精确组件均优先于 `emotion_buffer`。
- 场景无叙事价值时，不设点优先于 `emotion_buffer`。

**P5 正例与反例**

- 正例：高强度朝堂冲突后切换到安静庭院。
- 正例：新角色首次登场并建立重要性格特征。
- 反例：反派被打脸，应使用 `shatter_strike`。
- 反例：无信息、无情绪的普通走路过场，应不设互动点。

---

## 2.12 `episode_end_prediction` 剧尾预测

![剧尾预测](effect-screenshots/11-episode-end-prediction.png)

**P1 核心语义定义**

固定用于集尾悬念，让观众对下一集或后续剧情进行有依据、可验证的预判。

**P2 适用场景正例**

- 集尾抛出明确 cliffhanger。
- 当前冲突尚未解决，最后一个镜头提出新问题。
- 判断标准：互动点位于集尾前后 12 秒内，并且问题能在后续剧情得到验证。

**P3 硬排除规则**

- 不在集尾前后 12 秒范围内时不得使用。
- 预测结果无法在后续剧情验证时不得使用。
- 没有任何线索支撑、纯靠猜测的问题不得使用。

**P4 选择优先级**

- 位于集尾规定范围且存在有效悬念：优先于 `prediction_card`。
- 集中段悬念：改用 `prediction_card`。
- 询问已有线索意义：改用 `clue_judge_card`。

**P5 正例与反例**

- 正例：神秘人出手救下主角，镜头切黑，询问其身份。
- 正例：角色打开密信但内容尚未展示，当前集结束。
- 反例：集中段角色面临选择，应使用 `prediction_card`。
- 反例：询问下一集是否会出现新人，缺少线索与验证价值。

## 3. 组件选择决策树

```text
1. 当前场景是情绪释放吗？
   - 解气、反击成功 -> shatter_strike
   - 不公、愤怒仍在积累 -> anger_release
   - 悲伤、心疼、感动 -> tear_resonance
   - 明确让人笑出声 -> laugh_burst

2. 当前场景需要观众表达立场吗？
   - 双方道义大致对等 -> team_cheer
   - 角色处于危险或弱势，观众想保护 -> guardian_shield
   - 双方道义不对等且观众愤怒 -> anger_release

3. 当前场景需要观众思考吗？
   - 接下来会怎样 -> prediction_card
   - 这条信息意味着什么 -> clue_judge_card

4. 当前场景是正向氛围吗？
   - 达成目标的纯粹喜悦 -> celebrate_confetti
   - CP 或准 CP 关系升温 -> sugar_storm
   - 击败对手后的解气 -> shatter_strike

5. 当前场景位于集尾前后 12 秒且有悬念吗？
   - 是 -> episode_end_prediction

6. 以上都不是：
   - 有明确叙事价值 -> emotion_buffer
   - 没有叙事价值 -> 不设互动点
```

## 4. 易混淆组件快速判断

| 组件组合 | 判断问题 |
|---|---|
| `shatter_strike` / `anger_release` | 愤怒还在积累，还是已经迎来报应和释放？ |
| `shatter_strike` / `celebrate_confetti` | 喜悦来自击败对手，还是单纯达成目标？ |
| `laugh_burst` / `emotion_buffer` | 普通观众会明确笑出声，还是只觉得荒诞、离谱？ |

| `tear_resonance` / `anger_release` | 观众主要是心疼委屈，还是愤怒于不公？ |

## 5. 互动点数据要求

每个互动点至少应包含：

```json
{
  "id": "ip_unique_id",
  "start_ms": 12000,
  "end_ms": 20000,
  "component": "guardian_shield",
  "title": "守护加持",
  "emotion": "guard",
  "priority": 0.8,
  "highlight_reason": "角色孤身面对强敌，观众产生保护欲",
  "config": {}
}
```

字段要求：

- `id`：单集内唯一。
- `start_ms`：语义成立的开始时间。
- `end_ms`：语义失效的结束时间。
- `component`：必须是本文定义的 12 个组件之一。
- `title`：面向运营和调试的中文名称。
- `emotion`：只作为辅助标签，不作为组件选择的第一依据。
- `priority`：同一时间多个候选点时用于确定显示顺序。
- `highlight_reason`：必须描述“为什么此处符合该组件语义”，不能只写剧情摘要。
- `config`：组件题目、选项、阵营等可配置数据。
