# 前端 UI/UX 交互重做方案

## 1. 结论

当前前端的视觉基调基本可继续沿用，但交互逻辑、信息架构、状态持久化和后端契约都不完整。主要问题不是某一个按钮样式，而是产品闭环断裂：

- 发现作品、查看详情、播放、互动、评论、收藏、继续观看、个人中心之间没有稳定的行为链路。
- 多个可见控件没有真实功能，或者跳转目标不符合用户预期。
- 播放器控制层、作品信息层、互动层、底部导航层互相遮挡，导致核心操作不可用。
- 用户行为没有可靠持久化，收藏、评论、播放进度、历史记录无法形成可复用资产。
- 前端已经写了一些接口调用，但能力只覆盖极小子集，评论、用户中心、观看进度和剧集管理仍然缺少完整的数据模型和 API 支撑。
- 多个源码文件出现中文乱码，影响错误提示、空状态、按钮文案和后续维护。

后续应该把这次改造视为“交互系统重建”，而不是零散修补。

## 2. 当前问题清单

### 2.1 首页 / 短视频流

涉及文件：`cinematic-drama-app-frontend-source/src/pages/HomePage.tsx`

- 首屏中央会出现暂停/继续播放按钮。当前逻辑是只要视频处于暂停状态且无互动层，就显示中央播放按钮；初次加载、浏览器自动播放失败、切换集数时都会触发这个状态。用户会误以为页面默认处于暂停异常状态。
- 右侧操作栏包含收藏、评论、星标入口，但星标实际跳转到 `/theater`。图标语义和跳转结果不一致，用户会理解成收藏、评分或作品详情。
- 作品信息区域已经有标题链接到详情页，但右侧星标跳剧场会制造“点击作品信息进入剧场”的混淆。需要统一作品卡片/标题/简介的点击目标。
- 收藏按钮是按整部剧 `drama.id` 维护，首页却展示多集视频流。用户可能以为收藏的是当前剧集，实际收藏的是作品。
- 评论面板只支持加载、发布和关闭，没有删除、编辑、举报、点赞、回复、失败重试等基础操作。用户发布后没有管理余地。
- 评论输入失败后保留内容是正确方向，但缺少“发送中/重试/失败原因”状态，用户不清楚操作是否完成。
- 首页进度条已有自定义 pointer 逻辑，但只更新前端视频时间和本地进度数组，没有保存到后端，也没有进入播放页后的衔接。
- 首页的视频流只选择第一部有剧集的作品作为 feed 数据源，不能承载多部剧的发现流。
- 互动层 `z-[45]`，控制层和文案层多处使用 `z-40` / `z-[60]` / `z-[70]`，层级靠数字硬压，缺少统一规则，容易继续出现遮挡。

### 2.2 播放页

涉及文件：`cinematic-drama-app-frontend-source/src/pages/PlayerPage.tsx`

- 进度条使用原生 `input[type=range]`，`onChange` 中直接 `seekTo`。同时视频 `onTimeUpdate` 继续调用 timeline sync，拖动过程没有 `isSeeking` 状态隔离，容易出现拖动后被播放时间刷新回去的体验。
- 进入播放页时，manifest 或 episode 变化会强制 `video.currentTime = 0`，`setCurrentMs(0)`。这直接导致每次从头播放。
- 没有读取观看进度，也没有周期性保存观看进度。
- 底部标题、简介、播放按钮、进度条所在区域是 `absolute bottom-0 z-40`，互动层是 `z-30`。当互动点出现在底部区域时，播放器信息层会挡住互动组件。
- 播放页只有“返回首页”和“选集”入口，缺少收藏、评论、更多、下一集、清晰度/倍速等用户预期控制。
- 当前时间以 manifest duration 为主，但实际视频 metadata 又会覆盖 duration。manifest 和真实视频时长不一致时，进度条状态可能不稳定。

### 2.3 评论系统

涉及文件：`cinematic-drama-app-frontend-source/src/data/social.ts`

- 前端只定义了 `loadComments` 和 `postComment`，没有 update/delete/report/like/reply。
- `CommentItem` 中有 `likeCount`，但 UI 不显示也不能操作。
- 评论缺少“当前用户是否作者”的字段，前端无法判断是否允许删除或编辑。
- 评论列表没有分页、排序、楼中楼、审核状态、被删除状态、敏感词失败提示。
- 评论发布后重新拉全量列表，短期可接受，但后续应该改为乐观插入 + 后台确认。

### 2.4 搜索页

涉及文件：`cinematic-drama-app-frontend-source/src/pages/SearchPage.tsx`

- 没有明确的搜索按钮。用户只能输入后看列表变化，缺少提交感和结果状态。
- “进入 AI 沉浸式搜索”视觉权重高于普通搜索，使 AI 成为主路径，但当前 AI 更像补充能力。
- 普通搜索是前端本地过滤 `loadDramas()` 返回的数据，不能搜索剧集、情节点、台词、互动点或内容理解结果。
- 热搜标签只是把 query 设置成固定词，没有真正的热度、历史或推荐逻辑。
- 空结果只给普通空状态，没有推荐改写、进入 AI 搜索、清除筛选等恢复路径。

### 2.5 剧场页

涉及文件：`cinematic-drama-app-frontend-source/src/pages/TheaterPage.tsx`

- 剧场顶部只突出一个 featured drama，下面“全部剧集”也只属于这一个 featured drama。用户自然会感觉剧场只能容纳一部剧。
- “猜你喜欢”直接映射所有 dramas，没有推荐依据，也没有分类、筛选、排序、更新状态。
- 当数据库只有一部剧时，页面没有清晰说明“当前内容库仅 1 部”，而是继续展示推荐结构，造成假功能感。
- 剧场承担了“发现”和“选集”两个目标，但信息架构没有拆清楚：目录、榜单、类型、最近更新、继续观看应分区。

### 2.6 详情页

涉及文件：`cinematic-drama-app-frontend-source/src/pages/DetailPage.tsx`

- CTA 永远是“立即观看第一集”，没有读取观看进度，也没有“继续观看第 N 集 x:xx”。
- 没有收藏状态、评论入口、分享、追剧/更新提醒。
- 选集格子只显示集数和时长，缺少已看、看到位置、当前选中、锁定/未上线状态。
- 底部固定 CTA 和底部导航之间层级紧张，后续增加评论、收藏、更多操作时会继续遮挡。

### 2.7 用户页

涉及文件：`cinematic-drama-app-frontend-source/src/pages/ProfilePage.tsx`、`cinematic-drama-app-frontend-source/src/data/profile.ts`

- 用户页只有资料卡和几个菜单，看起来像装饰页。
- 继续观看只有单条 `continueWatching`，且没有进度百分比、时间点、封面、最近观看时间。
- “我的收藏”“互动记录”“离线缓存”都跳转到 `/theater`，属于典型假功能。
- 通知和设置只是 toast 文案，没有真实页面或禁用态。
- 统计数据缺少可追溯列表支撑，用户看到数字但不能进入明细。
- 离线缓存当前没有实现，不应该作为可点击入口长期存在。

### 2.8 文案与可维护性

多个 `.tsx` 和 `.ts` 文件中的中文文案出现乱码，例如加载、错误、按钮和空状态文本。必须统一编码和文案源，否则前端体验会持续暴露“半成品感”，也会让错误排查更困难。

## 3. 重做原则

### 3.1 先定义主路径

新的前端应围绕以下主路径组织：

1. 发现作品：从首页 feed、剧场、搜索进入作品。
2. 理解作品：进入详情页查看简介、选集、标签、热度、互动点摘要。
3. 播放作品：播放页支持稳定播放、拖动、互动、收藏、评论、下一集。
4. 沉淀行为：观看进度、收藏、评论、互动事件、搜索历史都被持久化。
5. 回到用户：用户页展示继续观看、收藏、历史、互动记录，让用户能接着做事。

### 3.2 每个可见控件必须有明确结果

所有按钮和入口只能处于三种状态之一：

- 可用：点击后完成真实操作或进入明确页面。
- 加载中：展示操作正在进行。
- 不可用：用 disabled 或“即将开放”状态明确告知，不跳转到无关页面。

禁止用 `/theater` 作为多个未实现功能的兜底跳转。

### 3.3 路由语义必须稳定

- 点击作品标题、封面、简介：进入 `/detail?drama=...`。
- 点击某一集播放：进入 `/player?drama=...&episode=...`。
- 点击剧场：进入 `/theater`，展示内容目录和推荐。
- 点击收藏列表：进入 `/profile/favorites` 或用户页内收藏 tab。
- 点击互动记录：进入 `/profile/interactions` 或用户页内互动 tab。
- 点击 AI 搜索：进入 `/ai`，作为搜索增强路径，不抢普通搜索主入口。

### 3.4 播放器优先保证手势稳定

播放器需要把“播放时间更新”和“用户拖动”分开：

- 拖动开始：设置 `isSeeking = true`，暂停 UI 进度自动同步。
- 拖动中：只更新预览时间和进度条视觉，不反复触发真实 seek。
- 拖动结束：提交一次 seek 到 video 和 timeline。
- 提交后：恢复 `onTimeUpdate` 同步。

这条原则同时适用于首页 feed 进度条和播放页进度条。

### 3.5 状态必须持久化

至少需要持久化：

- 收藏状态。
- 评论列表及当前用户评论操作。
- 观看进度，包括 drama、episode、positionMs、durationMs、updatedAt、completed。
- 互动事件和聚合统计。
- 搜索历史。

前端可以短期用 `localStorage` 做兜底，但最终权威状态必须来自 Django 后端。

## 4. 目标信息架构

### 4.1 首页 `/home`

定位：沉浸式发现流，不承担完整剧场目录。

应包含：

- 多部剧混排 feed，而不是只取第一部剧的所有 episode。
- 右侧操作栏：收藏、评论、分享、更多、详情。
- 作品信息区：标题和简介只进入详情页。
- 明确的播放/暂停状态，中央播放按钮只在用户主动暂停或自动播放失败时出现。
- 底部进度条可拖动、可预览、可保存。
- 评论面板为底部抽屉，支持发布、删除自己的评论、点赞、举报、失败重试。
- 互动组件出现时，播放器控制层自动让出空间，或互动层获得更高操作优先级。

### 4.2 播放页 `/player`

定位：完整观看和互动。

应包含：

- 进入时读取观看进度，展示“从上次位置继续 / 从头播放”。
- 每 5 秒、暂停、切集、退出页面、播放结束时保存进度。
- 自定义进度条，避免原生 range 和视频 timeupdate 抢状态。
- 底部控制栏可折叠，播放几秒后弱化；点击屏幕再显示。
- 互动层激活时，控制栏避让或暂时隐藏，保证互动按钮可点击。
- 操作区包括播放/暂停、下一集、选集、收藏、评论、分享。
- 当前集结束时进入“下一集 / 重播 / 返回详情”状态。

### 4.3 详情页 `/detail`

定位：作品详情和选集入口。

应包含：

- 作品基本信息、标签、简介、热度。
- 收藏/取消收藏。
- 主 CTA：如果有进度，显示“继续观看第 N 集”；否则显示“从第 1 集开始”。
- 选集网格显示：已看、看到位置、当前进度、未上线。
- 评论入口和精选评论。
- 从详情进入播放时传递准确 episode，并由播放页读取进度。

### 4.4 剧场 `/theater`

定位：内容库和推荐目录。

应包含：

- 顶部分类 tab：全部、最近更新、热门、收藏最多、题材分类。
- 多部剧卡片列表，不再只围绕 featuredDrama 组织。
- featured 区可保留，但不能让“全部剧集”只属于 featured。
- 当只有一部剧时，展示真实空态：“当前内容库仅 1 部，推荐区将在更多作品入库后启用”。
- 支持按题材、状态、集数范围、更新时间筛选。
- 每张剧卡进入详情页，而不是直接混用选集页。

### 4.5 搜索 `/search`

定位：普通搜索是主路径，AI 是增强能力。

应包含：

- 搜索输入框 + 明确的搜索按钮。
- 支持 Enter 提交。
- 搜索前展示最近搜索、热门搜索、题材快捷入口。
- 搜索后展示结果数量、清除按钮、排序或筛选。
- 普通搜索结果覆盖剧名、简介、题材、剧集标题。
- AI 搜索作为次级按钮，例如“用 AI 深度理解当前问题”。
- 空结果提供改写建议和 AI 搜索入口，而不是直接结束。

### 4.6 AI 搜索 `/ai`

定位：自然语言理解和内容解释，不替代普通搜索。

应包含：

- 允许用户输入更复杂的问题，例如“找女主反击最爽的一集”。
- 结果明确区分作品、剧集、片段、互动点。
- 每条结果说明匹配原因。
- 可跳转到详情页或指定播放时间点。

### 4.7 用户页 `/profile`

定位：个人行为资产中心。

应包含：

- 继续观看列表，至少展示最近 5 条。
- 我的收藏列表，可进入详情或取消收藏。
- 观看历史列表，可清空单条或全部。
- 我的评论列表，可跳转、删除。
- 互动记录列表，可查看参与过的互动点。
- 设置和通知如果未实现，应禁用或隐藏，不应假跳转。
- 离线缓存未实现前移除入口。

## 5. 后端/API 依赖

这次 UX 重做不能只靠前端完成。Django 后端需要至少提供以下接口能力。

### 5.1 用户与设备

- `GET /api/users/me/profile`
- `PATCH /api/users/me/profile`
- `GET /api/users/me/summary`

返回内容应包括用户展示信息、统计、最近观看、收藏数量、评论数量、互动数量。

### 5.2 收藏

- `GET /api/users/me/favorites`
- `PUT /api/users/me/favorites/{dramaId}`
- `DELETE /api/users/me/favorites/{dramaId}`

收藏结果需要返回当前状态和收藏总数，便于前端做乐观更新后的校正。

### 5.3 观看进度

- `GET /api/users/me/progress`
- `GET /api/users/me/progress/{dramaId}/{episodeNumber}`
- `PUT /api/users/me/progress/{dramaId}/{episodeNumber}`
- `DELETE /api/users/me/progress/{dramaId}/{episodeNumber}`

核心字段：

- `dramaId`
- `episodeNumber`
- `positionMs`
- `durationMs`
- `progressRatio`
- `completed`
- `updatedAt`

### 5.4 评论

- `GET /api/comments?drama=...&episode=...`
- `POST /api/comments`
- `PATCH /api/comments/{id}`
- `DELETE /api/comments/{id}`
- `POST /api/comments/{id}/like`
- `DELETE /api/comments/{id}/like`
- `POST /api/comments/{id}/report`

评论返回字段应包括：

- `id`
- `deviceId` 或 `userId`
- `displayName`
- `content`
- `likeCount`
- `likedByMe`
- `ownedByMe`
- `createdAt`
- `updatedAt`
- `status`

### 5.5 剧目目录

- `GET /api/dramas`
- `GET /api/dramas/{dramaId}`
- `GET /api/dramas/{dramaId}/episodes`
- `GET /api/feeds/home`
- `GET /api/theater/sections`

首页 feed 不应由前端自行选择第一部剧。后端应下发 feed items，字段包含 drama、episode、推荐理由、排序权重。

### 5.6 搜索

- `GET /api/search?q=...`
- `GET /api/search/suggestions?q=...`
- `GET /api/search/trending`
- `POST /api/ai/search`

普通搜索应基于数据库和内容理解索引；AI 搜索负责语义理解和解释。

### 5.7 行为事件

- `POST /api/events`
- `POST /api/events/batch`
- `GET /api/users/me/interactions`

事件队列可以保留，但需要服务端落库和去重策略，不能只写运行时 JSON。

## 6. 分阶段改造计划

### Phase 0：止损和语义修正

目标：先移除最伤体验的假功能和错误跳转。

- 修复所有中文乱码。
- 首页右侧星标入口改为“详情”或“更多”，不再跳剧场。
- 用户页未实现入口改为禁用或隐藏，不再统一跳剧场。
- 搜索页增加搜索按钮和 Enter 提交。
- 详情页 CTA 文案从固定“立即观看”改为可承载进度的组件。
- 明确所有加载、错误、空状态。

验收：

- 页面上没有点击后跳到无关页面的按钮。
- 主要中文文案全部正常显示。
- 搜索框旁有明确提交按钮。

### Phase 1：播放器和观看进度

目标：解决进度条、遮挡和从头播放问题。

- 播放页引入 `isSeeking` 状态。
- 自定义播放器进度条，拖动中只更新 preview，结束时提交 seek。
- 首页 feed 进度条复用同一套 seek 行为。
- 增加观看进度数据层：读取、保存、删除。
- 播放页进入时恢复上次位置。
- 每 5 秒、暂停、pagehide、ended 保存进度。
- 互动层激活时底部控制栏避让。

验收：

- 拖动进度条不会自动回弹到初始位置。
- 重新进入同一集时恢复到上次观看位置，误差不超过 2 秒。
- 互动点出现在底部时，操作按钮不会被标题、简介或进度条遮挡。

### Phase 2：评论、收藏、详情闭环

目标：让用户行为可进入、可管理、可回看。

- 收藏状态在首页、详情页、用户页一致。
- 评论支持发布、删除自己的评论、点赞、举报。
- 评论失败时保留输入内容，并提供重试。
- 详情页显示收藏、继续观看、选集进度。
- Profile 增加收藏列表和评论列表。

验收：

- 收藏后刷新仍然保持状态，并在用户页出现。
- 评论发布后可见，自己的评论可删除。
- 详情页能直接继续上次观看。

### Phase 3：剧场和搜索重构

目标：把内容发现从“单剧展示”改为真实目录和检索。

- 剧场按 sections 渲染：最近更新、热门、题材、全部。
- 只有一部剧时展示真实库存状态。
- 搜索页接入后端搜索接口。
- AI 搜索降为增强入口，并能接收普通搜索 query。
- 搜索结果支持作品、剧集、片段三类展示。

验收：

- 剧场可以展示多部剧和多个分类。
- 普通搜索可以通过按钮提交，并返回后端结果。
- AI 搜索不再压过普通搜索主路径。

### Phase 4：行为数据和运营能力

目标：把互动、观看和推荐变成可迭代资产。

- 行为事件批量上报并落库。
- 用户页展示互动历史。
- 首页 feed 从后端推荐接口获取。
- 推荐区不再使用硬编码“猜你喜欢”。
- 增加基础埋点：曝光、播放、暂停、seek、收藏、评论、详情点击、搜索提交。

验收：

- 后端可查询每个用户的观看、收藏、评论、互动记录。
- 首页 feed 不依赖前端选择第一部剧。
- 推荐和热搜至少有真实数据来源或明确空态。

## 7. 前端组件重构建议

建议新增或拆分以下组件：

- `VideoProgressBar`：统一首页和播放页 seek 行为。
- `PlayerControls`：播放页底部控制层，支持折叠和避让。
- `InteractionLayerHost`：统一互动层挂载、层级和避让信号。
- `CommentSheet`：评论抽屉，支持列表、发布、删除、点赞、举报。
- `FavoriteButton`：收藏按钮，封装乐观更新、回滚、加载态。
- `ContinueWatchingCard`：继续观看卡片，用于详情页和用户页。
- `EpisodeGrid`：选集网格，支持已看、进度、当前集状态。
- `DramaCard`：剧场、搜索、收藏列表复用。
- `SearchBar`：输入、按钮、Enter、清空、加载状态。
- `ProfileSection`：用户页收藏、历史、评论、互动记录分区。

组件拆分的关键不是抽象样式，而是统一行为：同一个收藏按钮、同一个进度条、同一个剧卡不能在不同页面表现出不同业务语义。

## 8. 数据流建议

短期：

- 前端增加 `watchProgress.ts`，先支持后端接口；接口不可用时用 `localStorage` 兜底。
- 收藏和评论继续使用 `social.ts`，但扩展 delete/like/report。
- 页面内保留乐观更新，但失败必须回滚或展示重试。

中期：

- 使用 Django API 作为权威状态。
- 首页 feed 改为读取 `/api/feeds/home`。
- Profile 改为读取多个真实列表，而不是单一 profile summary。

长期：

- 将搜索、推荐、互动统计与内容理解结果打通。
- 前端不再关心文件扫描或本地 manifest 细节，只消费稳定 API。

## 9. 验收标准

这次重做完成后，至少应满足以下标准：

- 用户打开首页，不会无故看到令人困惑的中央暂停按钮。
- 点击作品标题、封面或简介，只进入作品详情页。
- 首页、详情页、播放页的收藏状态一致，刷新后不丢失。
- 播放页进度条可拖动，拖动后不会回弹。
- 关闭页面再回来，同一集从上次进度继续播放。
- 评论可以发布、展示、删除自己的评论；发布失败时内容不丢。
- 用户页不是装饰页，至少能进入继续观看、收藏、评论或历史中的两个真实列表。
- 剧场能展示多部剧；当只有一部剧时，页面明确说明内容库状态。
- 搜索页有普通搜索按钮，AI 搜索是辅助入口。
- 所有可见按钮都能完成真实动作，或明确处于不可用状态。
- 播放器标题、简介、控制条不会遮挡互动组件和核心操作。
- 页面文案无乱码。

## 10. 优先级建议

最高优先级：

1. 修复文案乱码。
2. 修复播放器进度条拖动和观看进度恢复。
3. 修复遮挡和错误跳转。
4. 移除或禁用假功能入口。

第二优先级：

1. 评论管理能力。
2. 收藏在详情页、首页、用户页的统一。
3. 用户页真实列表。
4. 剧场多剧目录。

第三优先级：

1. 后端搜索。
2. AI 搜索结果解释和跳转到片段。
3. 行为数据统计和推荐。

这个顺序能最快把“能不能用”的问题解决，再逐步补齐“好不好用”和“能不能运营”的问题。
